# Context Bridge MCP — CLAUDE.md

## Project Overview

This is a local MCP (Model Context Protocol) server called **context-bridge-mcp**. Its purpose is to
save and restore Claude session context across Claude Chat, Claude Code, and Claude Cowork via a
shared local SQLite database.

One command saves the current session. Another retrieves it. Any Claude tool that has this MCP
registered can read from or write to the same store.

---

## Current Phase: Phase 1

### In Scope
- SQLite storage only
- Three MCP tools: `save_context`, `load_context`, `list_contexts`
- Auto-extraction of facts and tags by Claude before saving
- Last 10 messages stored as raw role/content pairs
- Announced (not silent) context loading
- Single user (username stored for future use)
- Works across Claude Chat, Claude Code, and Claude Cowork via Claude Desktop

### Out of Scope (Do Not Build Yet)
- CLAUDE.md write-back on load → Phase 2
- JSON export/import → Phase 2
- Supabase/Postgres adapter → Phase 3
- Multi-user logic → Phase 3
- Overwrite/versioning behavior → Phase 2

## Database Schema

Single table `contexts`:

```sql
CREATE TABLE IF NOT EXISTS contexts (
  id            TEXT PRIMARY KEY,        -- UUID v4
  title         TEXT NOT NULL,
  username      TEXT NOT NULL,           -- single user for now, stored for future
  source_tool   TEXT NOT NULL,           -- "chat" | "code" | "cowork"
  tags          TEXT NOT NULL,           -- JSON array string, auto-generated
  facts         TEXT NOT NULL,           -- JSON object: task, decisions, open_questions, entities
  last_messages TEXT NOT NULL,           -- JSON array of {role, content} pairs, last 10
  created_at    DATETIME NOT NULL,
  updated_at    DATETIME NOT NULL
);
```

The `facts` JSON object structure:
```json
{
  "current_task": "string",
  "key_decisions": ["string"],
  "open_questions": ["string"],
  "entities": ["string"]
}
```

## MCP Tools

### `save_context`

**What it does:** Persists the current session context to SQLite.

**Inputs:**
```
title       string   — human-readable name for this context
username    string   — user identifier (single user for now)
source_tool string   — "chat" | "code" | "cowork"
facts       object   — auto-extracted by Claude before calling this tool
tags        string[] — auto-generated by Claude before calling this tool
messages    array    — last 10 messages as raw {role, content} pairs
```

**Behavior:**
- Always inserts a new record (append-only, never overwrites)
- Generates a UUID v4 as the `id`
- Returns the saved `id` so it can be referenced later

**Important:** Claude must perform the extraction step before calling this tool.
The caller (Claude) is responsible for generating `facts` and `tags` in the same
internal pass prior to invoking `save_context`.

Facts extraction prompt Claude should use internally:
> "Summarize this session into: current_task (one sentence), key_decisions (list),
> open_questions (list), entities (names, variables, files, services in play).
> Also generate 3-5 short tags describing this session's topic."

**Annotations:** `readOnlyHint: false`, `destructiveHint: false`, `idempotentHint: false`

---

### `load_context`

**What it does:** Retrieves a saved context and announces it in the conversation.

**Inputs (one required):**
```
id      string?   — exact UUID lookup
title   string?   — fuzzy title search (LIKE %title%)
tag     string?   — filter by a single tag
latest  boolean?  — if true, returns the most recently created record
```

**Behavior:**
- Retrieves matching record (if multiple match title/tag, returns the most recent)
- Claude announces the loaded context visibly in the conversation, never silently:
  - Title and source tool
  - When it was saved
  - Facts summary (task, decisions, questions, entities)
  - Last messages rendered as a condensed transcript
- Does NOT write to CLAUDE.md or system prompt (Phase 2)

**Announcement format Claude should use:**
```
📂 Loaded context: "{title}"
Saved from: {source_tool} on {created_at}

Current task: {facts.current_task}
Key decisions: {facts.key_decisions}
Open questions: {facts.open_questions}
Entities in play: {facts.entities}

--- Last messages ---
{condensed transcript of last_messages}
```

**Annotations:** `readOnlyHint: true`, `destructiveHint: false`, `idempotentHint: true`

---

### `list_contexts`

**What it does:** Returns a browsable list of saved contexts.

**Inputs (all optional):**
```
tag         string?  — filter by tag
source_tool string?  — filter by "chat" | "code" | "cowork"
limit       number?  — default 20
```

**Behavior:**
- Returns a readable table of records sorted by `created_at` descending
- Columns: id (short — first 8 chars), title, source_tool, tags, created_at

**Annotations:** `readOnlyHint: true`, `destructiveHint: false`, `idempotentHint: true`

## Key Conventions

- **Always append-only** — never delete or overwrite records in Phase 1
- **Facts and tags are Claude's responsibility** — the MCP tool stores what Claude passes; it does not extract
- **Fuzzy title search** — use SQL `LIKE %query%` for title matching in `load_context`
- **Multiple matches** — always return the most recently created record when ambiguous
- **DB directory** — auto-create `~/.context-bridge/` on first run if it doesn't exist
- **Error messages** — must be actionable (e.g., "No context found with title matching 'auth flow'. Try list_contexts to browse available records.")
